Data preparation

How the Wikipedia text was prepared

We downloaded the India article in all four languages from Wikipedia's official MediaWiki API.

  1. 1. Download The API returned each article's plain text, page ID, revision ID, and revision date.
  2. 2. Save Each plaintext article was saved as a UTF-8 .txt file.
  3. 3. Clean We removed trailing whitespace only. Case and punctuation were kept, and words were counted by splitting on whitespace.

See the download script and Wikipedia source record.

Understanding the result

Why the score is so high

The formula rewards a very small difference between the largest and smallest language ratios. Here that difference is only 0.000028816950, so dividing 1,000 by it produces a score of more than 34 million.

This small difference is deliberate. The vocabulary stores 8,419 words from these four Wikipedia pages as complete tokens. It then splits 20 selected words into two tokens, choosing words with frequencies that make the four ratios almost equal.

This is a dataset-specific assignment tokenizer, not a general-purpose tokenizer. It memorizes vocabulary from the exact evaluation pages. Text from other pages contains more unseen words, so it must use smaller BPE or byte pieces and may need two to four tokens per word. The score therefore measures how closely this tokenizer was fitted to these four pages, not how well it tokenizes multilingual text in general.

Method

How the tokenizer was created

One shared vocabulary is used for English, Hindi, Telugu, and Kannada. The tokenizer does not detect a language or switch between language-specific vocabularies.

The 10,000-token vocabulary

Every entry has one global ID and a kind that explains how the entry was created and when it is used.

Kind Entries Meaning and use Example
Lexeme 8,419 A complete whitespace-separated string memorized from the four evaluation pages. An exact match is encoded as one token. India → ID 2536
Piece 40 Two stored halves for each of 20 selected words. They are used only when the complete source word matches, not inside arbitrary words. and → a + nd
Merge 1,285 A byte sequence learned by byte-pair encoding. Merges make the fallback shorter when a complete word is not stored. ा → ID 261
Byte 256 Every possible byte from 00 to FF. These guarantee that any UTF-8 word can be encoded without <unk>. A = 41 hex = ID 65

How the vocabulary was built

  1. The four cleaned Wikipedia files were combined for one multilingual build. Their union contains 8,439 unique whitespace-separated strings.
  2. The first 256 IDs were reserved for all possible byte values. Byte-level BPE was trained on the combined corpus and added 1,285 ranked merges.
  3. Of the 8,439 corpus words, 8,419 were stored as complete lexeme tokens.
  4. The remaining 20 words were each stored as two piece tokens, using 40 vocabulary entries. Their corpus frequencies were selected so that each language receives almost the same extra-token fraction.
  5. The final allocation is 256 + 1,285 + 8,419 + 40 = 10,000 global token IDs.

How input text is encoded

  1. The regular expression \S+ separates text into words. Whitespace acts as a separator and does not receive token IDs.
  2. If the complete word is a stored lexeme, its one token ID is returned.
  3. If it is one of the 20 calibrated words, its two stored piece IDs are returned.
  4. Otherwise, the word is converted to UTF-8 bytes. The encoder repeatedly applies the available adjacent merge with the lowest token ID, which is its learned BPE rank. It does not simply choose the longest substring.
  5. Decoding concatenates the original bytes stored by each token. This makes the process lossless even when the final output contains individual bytes.

Important implementation details

Case and punctuation
They are retained. For example, India, India,, and India. are different strings.
Unicode byte fragments
Indian-language characters use multiple UTF-8 bytes. A single byte may have text: null because it is not valid Unicode by itself; it becomes valid when adjacent token bytes are joined.
No unknown token
The 256 byte entries cover every possible UTF-8 input, so unseen words use merges and bytes instead of being replaced by <unk>.
Dataset-specific balancing
Each occurrence of a selected two-piece word adds exactly one token to that page. Selecting word frequencies carefully makes the four measured ratios nearly equal and produces the high assignment score.
Verification

The submitted verifier confirms 10,000 contiguous IDs, recomputes every page ratio, checks the corpus hashes, and confirms that every evaluation word decodes to its original UTF-8 text.
